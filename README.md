# CI-Project

Project repository for the Computational Imaging Group B assignment.

The project studies Sparse-views CT reconstruction on the Mayo dataset. The required setup follows the project specifications adopted for this repository:

- resize images to 256x256;
- generate sparse-view CT observations with 180, 90, 60, and 45 angles;
- add noise level 0.005;
- compare all methods on the same degraded inputs;
- report PSNR, SSIM, visual comparisons, and discussion.

For a two-student group, this repository is planned around three methods:

1. Total p-Variation regularization with p in {0.1, 0.5};
2. an end-to-end UNet reconstructor;
3. DiffPIR adapted to Sparse-views CT.

The hybrid PD-Net method is intentionally excluded, as allowed by the
specifications for groups of two students.

## Repository Layout

```text
configs/   Experiment and path configuration.
src/       Project code modules.
scripts/   Executable scripts for each pipeline step.
notebooks/ Optional exploration and Colab notebooks.
outputs/   Generated metrics, figures, and reconstructions.
docs/      Project notes, source tracking, and presentation planning.
```

Large data, trained weights, external repositories, and generated outputs are not tracked by Git.

## External Layout

The local project folder is expected to have this structure:

```text
Progetto/
├── CI-Project/      # this repository
├── Mayo/            # original Mayo dataset, outside Git
│   ├── train/
│   └── test/
├── data/
│   └── processed/   # generated clean/degraded tensor shards, outside Git
├── weights/         # trained model weights, outside Git
└── external/        # optional external code, outside Git
```

On Google Drive, the project root currently follows this layout:

```text
/content/drive/MyDrive/LM_INFORMATICA/COMPUTATIONAL_IMAGING/
├── Mayo/
│   ├── train/
│   └── test/
├── IPPy/
├── data/
│   └── processed/
├── checkpoints/
├── weights/
├── outputs/
└── external/
```

The Mayo dataset is used with the split already provided by the course material:
`train/` for model fitting and `test/` for final evaluation. No validation split
is generated in `01_data_and_degradation.ipynb`, because the course homework
pipelines use the same train/test convention.

The processed data generated by `notebooks/01_data_and_degradation.ipynb` are
saved as PyTorch `.pt` shards under:

```text
data/processed/
├── manifest.json
├── train/
│   └── Cxxx/
│       └── Cxxx_shard_0000.pt
└── test/
    └── C081/
        └── C081_shard_0000.pt
```

Each shard stores a dictionary with the following loading contract:

```python
payload = torch.load(shard_path, map_location="cpu")
clean = payload["clean"]                  # [B, 1, 256, 256]
sinogram_180 = payload["sinograms"]["180"] # [B, 1, 180, 256]
sinogram_90 = payload["sinograms"]["90"]   # [B, 1, 90, 256]
sinogram_60 = payload["sinograms"]["60"]   # [B, 1, 60, 256]
sinogram_45 = payload["sinograms"]["45"]   # [B, 1, 45, 256]
source_paths = payload["source_paths"]
metadata = payload["metadata"]
```

All reconstruction notebooks must load these saved tensors instead of
regenerating degraded measurements, so that TpV, UNet, DiffPIR, and the final
comparison use identical inputs.

Filesystem paths are tracked in `configs/paths.yaml` and
`configs/paths_colab.yaml`; Colab notebooks may still define the same Drive root
explicitly for readability.

## First Check

After placing the dataset in `../Mayo`, verify the structure with:

```bash
python3 scripts/inspect_dataset.py
```

On Colab, after mounting Google Drive and cloning the repository, use:

```bash
python scripts/inspect_dataset.py --paths configs/paths_colab.yaml
```
